# 权限角色系统规划

> 当前 v2 已上线，本文档记录 v3 演进方案。按需实施。

---

## 一、当前架构（v2）

### 1.1 权限链路

```
角色(Role) ──关联──→ 菜单(MenuItem) ──绑定──→ 接口权限(permissionCodes)
                                                        ↓
                                              角色有效权限 = 所有关联菜单的 permissionCodes 并集
```

### 1.2 数据模型

```
┌─────────────┐     ┌──────────────┐     ┌───────────────────┐
│    Role      │────→│  RoleMenuItem │────→│    MenuItem       │
│              │     │  (关联表)      │     │                   │
│ permissionCodes[]  │  roleId       │     │ permissionCodes[] │
│ (运行时只读)  │     │  menuItemId   │     │ (菜单绑定的权限)   │
└─────────────┘     └──────────────┘     └───────────────────┘
```

- `Role.permissionCodes`：单一权限来源（v2 根解，由菜单勾选自动派发）
- `MenuItem.permissionCodes`：菜单绑定的接口权限码
- `RolePermission` 表：已删除（v2，双源合一）

### 1.3 现有角色模板

| 模板 | 权限数 | 覆盖场景 |
|------|--------|----------|
| full（全权） | 42 | 超级管理员 |
| sales（销售策划） | 21 | 意向单 + 合同 |
| design（设计执行） | 32 | 项目 + 场景 + 物料 |
| ops（运营统筹） | 22 | 项目 + 时间线 + 物料 |
| readonly（只读） | 15 | 只读查看 |

### 1.4 局限性

| 问题 | 影响 |
|------|------|
| 权限粒度由菜单决定 | "只读项目"需要单独建菜单项 |
| 菜单承担双重职责 | 既是 UI 入口，又是权限来源 |
| 权限码对策划师不可见 | 无法直接给角色分配特定权限 |
| API-only 访问无菜单概念 | 移动端 / 第三方集成时菜单无意义 |

## 二、v3 架构方案：权限组（Permission Group）

### 2.1 核心思想

**权限绑角色，菜单只管 UI。**

```
v2: 角色 ──→ 菜单 ──→ 权限（间接，粒度粗）
v3: 角色 ──→ 权限组 ──→ 权限（直接，精细控制）
      └──→ 菜单（仅控制 UI 显示，不再派发权限）
```

### 2.2 数据模型

```prisma
// 新增：权限组（给策划师看的"能力包"）
model PermissionGroup {
  id              String   @id @default(cuid())
  tenantId        String?
  scope           RoleScope
  code            String   // 如 "project.readonly", "project.full", "contract.full"
  name            String   // 如 "项目管理-只读", "项目管理-全权", "合同管理-全权"
  description     String?
  permissionCodes String[] // 组内包含的具体权限码
  isBuiltIn       Boolean  @default(false)
  createdAt       DateTime @default(now())

  tenant Tenant? @relation(fields: [tenantId], references: [id], onDelete: Cascade)
  roles  RolePermissionGroup[]

  @@unique([scope, tenantId, code])
  @@index([tenantId])
  @@map("permission_groups")
}

// 新增：角色 ↔ 权限组关联
model RolePermissionGroup {
  roleId             String
  permissionGroupId  String

  role            Role            @relation(fields: [roleId], references: [id], onDelete: Cascade)
  permissionGroup PermissionGroup @relation(fields: [permissionGroupId], references: [id], onDelete: Cascade)

  @@id([roleId, permissionGroupId])
  @@map("role_permission_groups")
}

// 修改：Role 删除 permissionCodes 字段，改用关联
model Role {
  // ... 其他字段不变
  // 删除: permissionCodes String[]
  // 新增:
  permissionGroups RolePermissionGroup[]
  menus            RoleMenuItem[]
}
```

## 三、权限组 vs 菜单绑定：对比

| 维度 | v2（菜单绑定） | v3（权限组） |
|------|---------------|-------------|
| 权限粒度 | 菜单级（粗） | 权限组级（精细） |
| "只读项目" | 需要新建菜单项 | 选 `project.readonly` 即可 |
| 策划师理解成本 | 低（选菜单） | 低（选权限组，中文名） |
| API-only 访问 | 菜单无意义 | 权限组有意义 |
| 菜单职责 | UI + 权限（双重） | 仅 UI（单一） |
| 权限变更传播 | 改菜单 → 影响所有角色 | 改权限组 → 影响关联角色 |
| 移动端/第三方 | 需要适配 | 直接用权限组 |
| 审计清晰度 | 需要反推菜单 | 直接看权限组 |

## 五、权限组设计原则

### 5.1 命名规范

```
{资源}.{操作级别}

示例：
project.readonly    项目-只读
project.full        项目-全权
project.create      项目-创建
scene.full          场景-全权
contract.readonly   合同-只读
```

### 5.2 粒度控制

| 操作级别 | 包含权限 | 适用场景 |
|----------|----------|----------|
| `readonly` | *.read | 只读查看 |
| `create` | *.read + *.create | 可创建不可编辑 |
| `full` | *.read + *.create + *.update + *.delete | 完全控制 |
| `manage` | *.read + *.manage | 管理操作（如审批） |

### 5.3 组合规则

- 一个角色可以关联多个权限组（取并集）
- 权限组可以跨资源组合（如 `project.readonly` + `scene.readonly`）
- 预置模板覆盖 90% 场景，自定义覆盖 10%

## 六、与现有系统的兼容性

### 6.1 向后兼容

- `role.permissionCodes` 字段保留（从权限组计算，只读）
- 现有 API 返回格式不变
- 现有 `ROLE_TEMPLATES` 映射为 v3 权限组

### 6.2 迁移路径

```
v2 角色（permissionCodes: ['project.read', 'project.create']）
  ↓ 自动匹配
v3 角色（permissionGroups: [project.create]）
```

迁移脚本：遍历所有角色，根据 `permissionCodes` 匹配最合适的权限组组合。

### 6.3 灰度策略

1. 先在测试租户创建 v3 角色
2. 验证权限计算正确
3. 迁移现有角色（脚本）
4. 删除 `Role.permissionCodes` 字段（最后一步）

## 七、决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 权限绑角色还是菜单 | 权限绑角色（通过权限组） | 业界标准，精细控制，职责分离 |
| 权限组粒度 | 资源级（project.readonly） | 平衡灵活性和复杂度 |
| 是否删除菜单权限 | 保留但不作为权限来源 | 向后兼容，菜单仍控制 UI |
| 预置模板数量 | 6 个（full/sales/design/ops/readonly/custom） | 覆盖 90% 场景 |
| 权限组是否支持租户自定义 | 是 | 租户可能有特殊权限需求 |

## 八、风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 迁移脚本匹配不准 | 角色权限错误 | 迁移前备份，灰度验证 |
| 权限组过多 | 策划师选择困难 | 预置模板 + 搜索/筛选 |
| 性能下降（多一次关联查询） | API 响应变慢 | 缓存权限组（Redis / in-memory） |
| 现有代码改动量大 | 引入 bug | 分阶段实施，每阶段完整测试 |
